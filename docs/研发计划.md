# 分解者（Decomposer）AI Skill 研发计划

## 一、项目概述

### 1.1 项目定位
- **Skill 名称**：分解者（Decomposer）
- **核心使命**：将"未知的未知"转化为"已知的未知"
- **两大核心特质**：诚实性 + 方法论性
- **反空壳原则**：绝不输出华丽但空洞的模块列表

### 1.2 项目现状
- 项目目录：`Decomposer Skill`（当前仅有 README.md、LICENSE、.gitattributes）
- 设计文档已完成四轮认知迭代（技术架构师视角 → 零代码执行者视角 → 决策卡策略家视角 → MIT学者视角）
- 处于"概念验证 → 工程实现"的临界点

### 1.3 目标用户画像
分解者不是为单一用户类型设计的，而是为**任何面临"庞大而模糊的工程"的人**提供分层拆解服务：

| 用户类型 | 核心痛点 | 对应拆解模式 |
|---------|---------|------------|
| 技术架构师/产品经理 | 怕空壳、怕假进度 | 不确定性热力图 + 动态节点 |
| 非技术创始人/零代码从业者 | 不会编码、没有研究时间 | 人工高仿 + 7天致死测试链 |
| 资源持有者/决策者 | 有方向但缺抓手 | 三张决策卡（致命前提/资源唤醒/MVS） |
| 研究型学者/科学家 | 研究问题堆叠、不确定性极高 | 可证伪性白皮书 + 层级化仿真 |

## 二、需求分析与定位校准阶段

### 2.1 阶段目标
将设计文档中的"思想实验"转化为可工程化的 Skill 需求规格，明确边界、优先级和验收标准。

### 2.2 核心任务

#### 任务 2.2.1：需求结构化梳理
- 系统梳理四轮认知迭代中的核心概念，建立概念图谱
- 识别重复概念和冲突概念，进行统一命名
- 输出《Decomposer 需求规格说明书 v0.1》

**关键产出概念清单**：
- 不确定性热力图（绿/黄/红区）
- 反空壳拆解法（动态节点 + 存亡条件）
- 边界穿刺测试
- 人工高仿替代系统
- 7天致死测试链
- 三张决策卡（致命前提验证卡 / 资本资源唤醒卡 / MVS定义卡）
- 可证伪性白皮书
- 层级化仿真（快/中/慢变量层）
- 误差预算表
- 数据坟场协议

#### 任务 2.2.2：Skill 触发条件与边界定义
- 明确什么场景下应该触发 Decomposer（vs 其他 Skill）
- 定义 Decomposer 的"不做什么"清单（诚实性的体现）
- 与现有 Skill 的协作关系界定（brainstorming、writing-plans、consulting-analysis 等）

**与邻近 Skill 的边界**：
| Skill | 关注点 | Decomposer 的差异化 |
|-------|--------|-------------------|
| brainstorming | 从想法到设计方案的澄清 | 不做设计，做"未知的显形"——把模糊的恐惧变成具体的问题 |
| writing-plans | 已有需求后的实施计划 | 不做执行计划，做"认知拆解"——把未知分层分级 |
| consulting-analysis | 研究报告的结构化输出 | 不输出报告，输出"决策坐标"——让用户知道该验证什么 |

#### 任务 2.2.3：用户工作流设计
- 设计 4 种典型用户路径（对应 4 种视角）
- 每种路径的输入、输出、交互步数
- 设计"诚实性熔断"机制——当项目过于庞大时，第一时间告知用户边界

#### 任务 2.2.4：定位校准对话
基于用户规则中"提交前向用户提问以校准定位"的要求，进行以下校准：

1. **目标用户优先级**：4 种视角中，哪一种是首要服务对象？（决定 MVP 的形态）
2. **输出形态偏好**：用户更希望得到"结构化文档"还是"对话式引导"？
3. **深度 vs 广度**：是先做深一种模式，还是同时铺开 4 种模式？
4. **与现有工作流的集成点**：Decomposer 的输出应该如何被下游 Skill 消费？

### 2.3 交付物
1. 《Decomposer 需求规格说明书 v0.1》
2. 《Decomposer 概念图谱》（Markdown + Mermaid 图）
3. 《用户工作流设计文档》（4 条路径）
4. 定位校准对话记录与结论

### 2.4 验收标准
- 所有核心概念均有明确定义和使用场景
- Skill 边界清晰，不与现有 Skill 功能重叠
- 4 条用户路径均可走通（逻辑层面）
- 用户确认定位校准结论

## 三、Skill 架构设计阶段

### 3.1 阶段目标
基于 TRAE Skill 体系，设计 Decomposer 的文件结构、核心模块和交互流程。

### 3.2 技术选型依据
TRAE Skill 的标准结构为：
- 核心文件：`SKILL.md`（YAML Front Matter + Markdown 指令体）
- 可选子文件：`references/`（子场景参考文档）、`scripts/`（辅助脚本）、`templates/`（模板文件）
- 触发机制：通过 description 中的关键词匹配用户意图

### 3.3 核心任务

#### 任务 3.3.1：SKILL.md 主体结构设计
设计 SKILL.md 的整体架构，包含：

**Front Matter 设计**：
```yaml
name: decomposer
description: >
  Decompose ambiguous, large-scale projects into structured uncertainty maps.
  Transforms "unknown unknowns" into "known unknowns" through honest,
  methodological breakdown. Triggers when users face complex projects with
  high uncertainty, need to identify hidden risks, or want to avoid "empty
  shell" plans. Covers technical architecture, zero-code execution, strategic
  decision cards, and research-grade falsifiability analysis.
```

**主体章节设计**：
1. 核心哲学与原则（诚实性协议 + 反空壳原则）
2. 触发场景识别（4 种用户类型的自动识别逻辑）
3. 第一级：诚实性熔断（不确定性热力图生成）
4. 第二级：模式路由（自动选择/用户选择拆解模式）
5. 模式一：技术架构师模式
6. 模式二：零代码执行者模式
7. 模式三：决策卡策略家模式
8. 模式四：MIT学者模式
9. 输出格式规范
10. 质量自检清单

#### 任务 3.3.2：四种拆解模式的详细设计

**模式一：技术架构师模式**
- 输入：技术项目描述
- 输出：不确定性热力图 + 动态节点骨架 + 存亡条件清单
- 核心步骤：
  1. 扫描项目，识别已知/未知边界
  2. 生成绿/黄/红三区热力图
  3. 构建动态节点（每个节点附存亡条件）
  4. 标记占位符节点及其破局任务
  5. 输出可执行骨架

**模式二：零代码执行者模式**
- 输入：产品方向 + 零代码约束
- 输出：人工高仿方案 + 7天致死测试链 + 手工SOP
- 核心步骤：
  1. 识别核心价值主张
  2. 设计全人工中台模拟方案
  3. 拆解 7 天致死测试链（每天的具体任务）
  4. 编写手工流程 SOP
  5. 定义"做不下去的信号"和"可以冲的信号"

**模式三：决策卡策略家模式**
- 输入：方向 + 目标 + 资源约束
- 输出：三张决策卡 + 高杠杆动作建议
- 核心步骤：
  1. 致命前提验证卡（识别最可能失败的单一原因 + 外包验证方案）
  2. 资本/资源唤醒卡（翻译技术不确定性为非技术人员可操作的问题）
  3. MVS 定义卡（最小可卖单位设计 + 定价建议 + 投放渠道）

**模式四：MIT学者模式**
- 输入：研究型产品方向 + 科学问题
- 输出：可证伪性白皮书 + 层级化仿真方案 + 误差预算表
- 核心步骤：
  1. 生成可证伪性白皮书（主定理 + 摧毁条件）
  2. 数学建模建议（不变量识别 + 理论边界）
  3. 层级化仿真设计（快/中/慢变量层解耦）
  4. 误差预算表（各模块误差上限）
  5. 数据坟场协议（失败后的数据价值提取方案）

#### 任务 3.3.3：文件结构设计
```
Decomposer Skill/
├── SKILL.md                      # 主入口，核心逻辑与路由
├── README.md                     # 用户可读介绍（已有，需完善）

├── references/                   # 子场景参考文档
│   ├── architect-mode.md         # 技术架构师模式详细指南
│   ├── zero-code-mode.md         # 零代码执行者模式详细指南
│   ├── decision-cards-mode.md    # 决策卡策略家模式详细指南
│   └── mit-scholar-mode.md       # MIT学者模式详细指南
├── templates/                    # 输出模板
│   ├── uncertainty-heatmap.md    # 不确定性热力图模板
│   ├── dynamic-nodes-skeleton.md # 动态节点骨架模板
│   ├── 7day-death-test.md        # 7天致死测试链模板
│   ├── decision-cards.md         # 三张决策卡模板
│   ├── falsifiability-whitepaper.md  # 可证伪性白皮书模板
│   └── error-budget-table.md     # 误差预算表模板
└── docs/                         # 研发文档
    ├── requirements.md           # 需求规格说明书
    └── concept-map.md            # 概念图谱
```

#### 任务 3.3.4：诚实性机制设计
这是 Decomposer 的核心差异化能力，需要专门设计：

**诚实性熔断机制**：
- 每次拆解的第一步，必须先输出"我能做到什么/不能做到什么"
- 对用户项目规模进行诚实评估，不夸大能力
- 明确标注每个结论的置信度等级

**反空壳校验清单**（自检用）：
1. 每个模块是否都有具体的存亡条件？
2. 是否存在"听起来很专业但无法验证"的描述？
3. 黄区和红区的比例是否合理？（如果全是绿区，说明在造假）
4. 是否明确标注了"我不知道"的部分？
5. 输出是否可以被直接行动检验？

### 3.4 交付物
1. `SKILL.md` 完整设计稿
2. 4 份子模式参考文档（草稿）
3. 6 份输出模板（草稿）
4. 文件结构设计说明

### 3.5 验收标准
- SKILL.md 结构符合 TRAE Skill 规范
- 4 种模式的输入输出定义清晰、逻辑自洽
- 诚实性机制可操作、可检验
- 模板结构完整，占位符明确

## 四、核心模块实现阶段

### 4.1 阶段目标
将设计稿转化为可运行的 Skill 代码，实现核心功能。

### 4.2 核心任务（按优先级排序）

#### 任务 4.2.1：SKILL.md 主体实现（最高优先级）
- 编写完整的 SKILL.md 文件
- 实现 Front Matter（name、description）
- 实现核心章节：触发识别、诚实性熔断、模式路由
- 实现质量自检机制

**关键实现要点**：
- description 中包含足够多的触发关键词，确保 LLM 能正确识别触发场景
- 使用硬约束标记确保关键规则不被绕过
- 每一步骤都有明确的"怎么做"和"怎么检验"

#### 任务 4.2.2：模式三：决策卡策略家模式实现（次高优先级）
- 编写 `references/decision-cards-mode.md`
- 实现三张决策卡的生成逻辑
- 编写 `templates/decision-cards.md`

**为什么先做模式三**：
- 模式三最贴近"资源持有者"这一高价值用户群
- 输出物最具体（三张卡），容易验证质量
- 与其他 Skill 的协作点最清晰（可对接 consulting-analysis）

**验收用例**：
- 输入："我想做一个AI驱动的个性化教育产品，但我不会编码"
- 输出：致命前提验证卡（1个最危险假设 + 验证方案）、资源唤醒卡（3个技术拷问问题）、MVS定义卡（100元可卖的最小交付物）

#### 任务 4.2.3：模式一：技术架构师模式实现
- 编写 `references/architect-mode.md`
- 实现不确定性热力图生成逻辑
- 实现动态节点骨架生成逻辑
- 编写 `templates/uncertainty-heatmap.md`
- 编写 `templates/dynamic-nodes-skeleton.md`

**验收用例**：
- 输入："我想做一个类似抖音的短视频平台"
- 输出应包含：绿区（视频上传、播放列表等基础功能）、黄区（推荐算法冷启动）、红区（亿级用户并发、内容审核成本）
- 每个节点附存亡条件

#### 任务 4.2.4：模式二：零代码执行者模式实现
- 编写 `references/zero-code-mode.md`
- 实现人工高仿方案设计逻辑
- 实现 7 天致死测试链生成逻辑
- 编写 `templates/7day-death-test.md`

**验收用例**：
- 输入："我想做一个社区团购平台，零代码，只有我一个人"
- 输出应包含：全人工模拟方案、7天每天的具体任务、成功/失败信号

#### 任务 4.2.5：模式四：MIT学者模式实现
- 编写 `references/mit-scholar-mode.md`
- 实现可证伪性白皮书生成逻辑
- 实现层级化仿真设计逻辑
- 编写 `templates/falsifiability-whitepaper.md`
- 编写 `templates/error-budget-table.md`

**验收用例**：
- 输入："我想研究用LLM做数学推理的产品化方向"
- 输出应包含：主定理 + 摧毁条件、快/中/慢变量层设计、误差预算分配

#### 任务 4.2.6：模板文件完善
- 所有模板文件填充完整的结构
- 每个模板都有清晰的使用说明
- 模板中包含示例内容，便于理解

### 4.3 实现顺序说明
采用"核心骨架 → 最高价值模式 → 其他模式"的渐进式实现策略：

1. 先做 SKILL.md 主体（路由 + 熔断 + 自检）
2. 再做模式三（决策卡）——最易验证、最高价值
3. 然后模式一（技术架构师）——最经典、最易理解
4. 然后模式二（零代码）——最具实操性
5. 最后模式四（MIT学者）——最复杂、用户群最窄

### 4.4 交付物
1. 完整可运行的 `SKILL.md`
2. 4 份 references 文档
3. 6 份模板文件
4. 更新后的 README.md

### 4.5 验收标准
- Skill 可被 TRAE 正确加载和识别
- 每种模式至少通过 3 个测试用例
- 反空壳自检清单可有效运作
- 输出物中诚实标注了不确定性比例
- README.md 清晰说明 Skill 的用途和使用方法

## 五、测试与迭代阶段

### 5.1 阶段目标
通过多维度测试验证 Skill 的有效性，发现问题并迭代优化。

### 5.2 核心任务

#### 任务 5.2.1：功能测试
- 对每种模式设计 5-10 个测试用例
- 测试用例覆盖：简单项目、中等复杂度项目、极端复杂项目
- 验证输出格式是否符合模板
- 验证诚实性机制是否生效

**测试矩阵**：
| 模式 | 简单项目 | 中等项目 | 复杂项目 | 边界情况 |
|------|---------|---------|---------|---------|
| 技术架构师 | 待办清单App | SaaS后台 | 短视频平台 | 用户说"什么都想做" |
| 零代码 | 手工记账 | 社区团购 | 教育平台 | 用户完全没资源 |
| 决策卡 | 副业想法 | 创业方向 | 颠覆式创新 | 用户方向极其模糊 |
| MIT学者 | 算法优化 | 研究工具 | 基础研究产品 | 问题无法证伪 |

#### 任务 5.2.2：诚实性专项测试
这是 Decomposer 的核心质量指标：

**测试维度**：
1. **不夸大测试**：输入极端复杂项目，验证是否诚实告知边界
2. **不确定性标注率**：统计输出中黄区+红区占比（不应为0%）
3. **可证伪性测试**：每个论断是否都可以被检验/反驳
4. **反空壳测试**：是否存在"听起来专业但无法落地"的模块

**评分标准**（诚实性得分，满分100）：
- 明确告知能力边界（25分）
- 不确定性热力图比例合理（25分）
- 每个结论有置信度标注（25分）
- 无空洞模块列表（25分）

#### 任务 5.2.3：用户体验测试
- 邀请目标用户进行试用
- 收集反馈：是否理解输出、是否觉得有用、是否愿意再次使用
- 迭代优化交互流程和输出格式

#### 任务 5.2.4：与其他 Skill 的集成测试
- 测试 Decomposer → brainstorming 的衔接
- 测试 Decomposer → writing-plans 的衔接
- 测试 Decomposer → consulting-analysis 的衔接
- 测试 Decomposer → html-report 的衔接

### 5.3 迭代策略
- 每发现一个问题，先判断属于哪个层级（提示词问题/逻辑问题/架构问题）
- 小问题直接修复，大问题记录并规划到下一轮迭代
- 维护一个《已知限制清单》，诚实告知用户 Skill 的能力边界

### 5.4 交付物
1. 测试用例集（每个模式 5-10 个）
2. 测试报告（功能测试 + 诚实性测试 + 集成测试）
3. 《已知限制清单 v1.0》
4. 迭代后的 Skill 代码（v1.0）

### 5.5 验收标准
- 所有测试用例通过率 ≥ 80%
- 诚实性得分 ≥ 80 分
- 至少与 2 个下游 Skill 完成集成验证
- 已知限制清单完整、诚实

## 七、风险与应对

### 7.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| SKILL.md 提示词工程复杂，LLM 理解不稳定 | 输出质量波动大 | 高 | 1. 使用明确的结构化指令；2. 添加自检步骤；3. 多用示例（few-shot） |
| 4 种模式差异太大，一个 SKILL.md 装不下 | 代码臃肿、逻辑混乱 | 中 | 1. 用 references/ 子文件拆分；2. 先做核心模式，其他模式渐进添加 |
| 诚实性机制难以执行（LLM 倾向于"什么都能做"） | 违背核心原则 | 高 | 1. 在 SKILL.md 开头强化诚实性规则；2. 输出格式强制要求不确定性标注；3. 添加自检清单 |

### 7.2 产品风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 用户不知道什么时候该用 Decomposer | 使用率低 | 中 | 1. description 中写清触发关键词；2. README 用对比案例展示价值；3. 提供"什么时候用"的判断指南 |
| 4 种模式定位模糊，用户选不对 | 体验差 | 中 | 1. 设计自动识别逻辑；2. 首次使用时引导用户选择；3. 每种模式有清晰的适用场景说明 |
| "诚实性"反而让用户觉得 Skill 没用 | 用户流失 | 中 | 1. 解释"诚实 ≠ 无能"，诚实是为了更精准地行动；2. 用成功案例展示诚实拆解的价值 |

### 7.3 范围蔓延风险
Decomposer 的概念很宏大，容易陷入"什么都想做"的陷阱。

**应对策略**：
- 严格按照 MVP 思路，先做好 1-2 种模式
- 维护《不做清单》，每加一个功能先问"这是核心使命吗？"
- 定期回归核心使命："把未知的未知变成已知的未知"

## 八、关键成功指标

### 8.1 功能性指标
- 4 种模式全部实现并通过测试
- 每种模式至少有 3 个高质量示例输出
- 与至少 2 个下游 Skill 完成集成

### 8.2 诚实性指标
- 输出中黄区+红区比例 ≥ 30%（对于复杂项目）
- 明确标注"我不知道"的部分 ≥ 1 处
- 每个结论都有置信度或验证方式
- 不存在无法检验的空洞描述

### 8.3 可用性指标
- 用户能在 3 分钟内理解 Skill 的用途
- 用户能在第一次使用时获得有价值的输出
- 输出格式清晰、易读、可直接使用
